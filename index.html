01 Abstract

This study replicates A-MEM (Xu et al., 2024), an agentic memory system for LLM agents inspired by the Zettelkasten method, and extends its ablation with four novel conditions: a no-memory baseline, isolated linking removal, isolated evolution removal, and a retrieve-k sweep (k=10, k=20). Evaluated on the LoCoMo long-context conversational QA benchmark, I find that (1) a no-memory GPT-4o-mini baseline outperforms all GPT-3.5-turbo memory conditions on aggregate F1 (0.502 vs. 0.366–0.415), revealing a critical model-quality confound in the original paper; (2) within a controlled GPT-3.5-turbo group, linking and evolution show modest isolated contributions; and (3) A-MEM's gains are not model-agnostic — they scale with the quality of the underlying LLM performing the memory operations. These findings suggest that future memory system evaluations must include a same-model no-memory baseline to isolate architectural gains from model-capability effects.

02 Background & Motivation
  • →LLM agents have no persistent memory across sessions — they forget everything between conversations.
  • →Existing systems (MemGPT, MemoryBank) rely on fixed, predefined structures — manually specifying when, what, and how to store memories.
  • →Graph databases (Mem0) add structure but are rigid — unable to adapt to novel relationship types that emerge organically.
  • →Gap: No prior system organizes memory autonomously as it learns, without hardcoded rules.
  • →A-MEM proposes agentic memory: constructs, links, and evolves itself via LLM calls, inspired by Zettelkasten.
  • →This study asks: which components actually drive performance — and does the architecture hold across model tiers?
03 Memory Note Lifecycle
① Store
Turn → keywords + context + tags generated by LLM. Embedded as 384-dim vector → ChromaDB.
↓
② Link
Top-k cosine neighbors retrieved. LLM decides which to link → builds knowledge graph for multi-hop retrieval.
↓
③ Evolve
Linked neighbors rewrite their context + tags given new memory. Store refines itself continuously.
↓
④ Retrieve & Answer
Query embedded → top-k + linked neighbors injected into prompt → LLM answers with ~1,200 token context.
Inspired by Zettelkasten — atomic, interconnected notes. No predefined rules; the LLM decides what to link and how to update.

11 Key Findings
1
No Memory (GPT-4o-mini) beats all memory conditions. Aggregate F1 of 0.502 vs. 0.366–0.415. On Yes/No questions, No Memory scores 0.847. Reveals a critical model-quality confound absent from the original paper.
2
A-MEM's gains are not model-agnostic — they scale with LLM quality. GPT-3.5-turbo likely does not generate high-quality links or evolved contexts. The paper's strong results likely require GPT-4-class models doing the memory operations.
3
k=20 consistently outperforms k=10 (+0.049 F1). Retrieval breadth matters — more retrieved memories provide more signal for multi-hop and temporal chains.
4
Linking and evolution show modest isolated effects at GPT-3.5 scale. No Linking: 0.387; No Evolution: 0.388; Full k=10: 0.366. k=20 with all components (0.415) still outperforms both ablations.
12 Limitations
!Small sample: 1–2 LoCoMo conversations (199–304 questions). High variance.
!Model mismatch: Group A on locomo2, Group B on locomo1 — confounded by data and model.
!API quota limits: Full-scale locomo10 runs not feasible — free-tier limits hit mid-run.
!No significance testing: Differences are point estimates without confidence intervals.
!Single benchmark: LoCoMo focuses on personal conversational history only.
!Paper ablation not replicated exactly: Original uses GPT-4o-mini full dataset; mine uses GPT-3.5 on a subset.
13 Future Work
HighRun same-model controlled ablation on GPT-4o-mini at full scale. This is the critical missing experiment. All four conditions on locomo10 would definitively determine whether A-MEM's architecture adds value beyond model quality.
HighReplicate with open-weight local models (Llama 3.1-8B, Mistral-7B). Local inference allows full locomo10 runs at zero marginal cost, enables proper statistical testing across 10 conversations.
MedExtend retrieve-k sweep to k=5, 30, 50 with cost-accuracy tradeoff curves. k=20 > k=10, but optimal k and diminishing returns are unknown.
MedEvaluate on domain-specific benchmarks beyond LoCoMo — coding assistants, customer support, medical QA where memory density is higher.
LongDesign a lightweight memory proxy to replace LLM-based link and evolution calls. Memory ops account for 80% of API spend but deliver marginal F1 gains at GPT-3.5 scale.
LongStudy memory decay and forgetting strategies. A-MEM stores all memories indefinitely. Evaluating forgetting schedules is essential for real-world deployment viability.
InfraInstrument memory-level metrics alongside QA metrics. Adding link precision, evolution coherence, and retrieval recall@k would let future work diagnose whether poor QA stems from bad memory construction or bad retrieval.
14 Conclusions
  • ★A-MEM's architecture is sound but not model-agnostic. Its reported gains are partly a function of LLM quality doing the linking and evolving.
  • ★The no-memory upper bound is a critical missing baseline. All future memory evaluations should include a same-model no-memory condition.
  • ★Retrieval breadth (k) is the most actionable lever. k=20 consistently beats k=10 — tune k first before adding architectural complexity.
  • ★Cost scales superlinearly with usage. At 5 LLM calls per question with a growing memory store, A-MEM is expensive for high-volume deployments.
  • ★Replication revealed important omissions: no no-memory baseline, no isolated single-component ablations, no model-tier sensitivity analysis.
